## 1. 목적과 배경

Sigil Survivor는 1인 개발 웹게임으로, 본 계획서는 **개발과 QA를 한 사람이
수행하는 환경에서 품질을 체계적으로 보장하기 위한 기준**을 정의한다.
핵심 과제는 두 가지다:

1. 버전이 올라갈 때마다(v0.1→v0.8, 총 8회) 기존 기능이 깨지지 않았음을
   짧은 시간 안에 반복 확인하는 것 (회귀)
2. 감으로 하던 밸런스 판단을 수치 근거로 바꾸는 것 (데이터 검증)

## 2. 테스트 대상과 범위

**포함 (In Scope)**

| 영역 | 세부 |
|---|---|
| 코어 게임플레이 | 이동/대시, 무기 3종(부메랑·검격·오브), 패시브 4종, 시길 드로잉 스킬 |
| 전투/성장 | 적 3종+보스, 레벨업/업그레이드 선택, 경험치 곡선 |
| 밸런스 데이터 | WEAPONS·ENEMIES·DIFFS 테이블 전 수치 (spec.json 기준) |
| 난이도 시스템 | 3단계 배율 적용, 클리어 기반 해금 체인 |
| UI 플로우 | 시작/일시정지/로비 복귀/게임오버/재시작 |
| 메타 시스템 | 글로벌 리더보드(Supabase), 코덱스/도감 |
| 그래픽/사운드 | 에셋 로딩·폴백, 스프라이트 상태 전환, BGM/SFX·음소거 |

**제외 (Out of Scope) — 사유 명시**

- 모바일/터치 환경: 데스크톱 웹 우선 설계, 별도 대응 전까지 보류
- 브라우저 크로스 매트릭스: Chrome 기준 검증, 타 브라우저는 스팟 체크만
- 서버 부하/보안 침투: Supabase 무료 티어 프로토타입 단계로 범위 외
- 국제화(i18n): 한국어 단일 언어

## 3. 테스트 전략 — "코드가 곧 명세"

이 게임은 밸런스 수치가 전부 데이터 테이블(`const WEAPONS = {...}` 등)에
모여 있는 데이터 주도 구조다. 이를 명세로 삼아 다음 파이프라인을 운영한다:

```
게임 소스 ─
                          └─③ TC 자동 생성→ testcases.csv
빌드 ─④ 스모크 자동화(Playwright 5종, 19.2초)→ 테스트 가능 판정
```

**자동/수동 분담 기준** — 반복 주기 × 판정 명확성:

| | 판정이 기계적 | 판정에 사람 필요 |
|---|---|---|
| **매 버전 반복** | 자동화 (수치 검증, 스모크, 정합성) | 수동 회귀 체크리스트 |
| **단발성** | 스크립트 일회 실행 | 탐색적 테스트 |

현재 TC 40건 = AUTO 15 + MANUAL 25. 자동화는 수동 테스트를 대체하지 않고,
반복 비용을 줄여 탐색적 테스트 시간을 확보하는 역할로 정의한다.

**회귀 운영 규칙 (핵심)**
- 버그 1건 수정 = 회귀 TC 1건 추가 (manual_cases.json에 누적)
- 매 버전 핸드오프 시: 스펙 재추출 → 불변식 전체 PASS → 스모크 통과 →
  변경점별 QA 체크 항목 문서화

## 4. 우선순위 기준

| 등급 | 정의 | 예시 |
|---|---|---|
| P0 | 진행 불가/데이터 훼손. 발견 즉시 수정 전까지 배포 차단 | 난이도 해금 오동작, 게임 시작 불가 |
| P1 | 핵심 경험 훼손. 당일 수정 목표 | 무기 수치 불일치, 조종사 스프라이트 소실(BUG-001) |
| P2 | 불편하지만 우회 가능 | 타일 이음새, 이펙트 과다 |
| P3 | 사소/개선 제안 | 문구, 폴리싱 |

심각도(영향 크기)와 우선순위(수정 순서)는 분리 판단한다 — 버그 리포트
템플릿에 두 필드를 별도로 둔 이유.

## 6. 시작·종료 조건 (Entry / Exit Criteria)

**테스트 시작 조건**: 스모크 5종 통과 (미통과 빌드는 본 테스트 진입 불가 —
"테스트할 수 없는 빌드에 시간을 쓰지 않는다")

**버전 배포(핸드오프) 종료 조건**:
- [ ] 불변식 검증 18건 전체 PASS (의도된 FAIL은 사유 기록 후 승인)
- [ ] 스모크 5종 통과 + 콘솔 에러 0건
- [ ] 신규/수정 기능의 회귀 TC 등록 완료
- [ ] 미해결 P0/P1 = 0건 (P2 이하는 알려진 이슈로 기록 후 배포 가능)
